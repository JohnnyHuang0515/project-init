# Anatomy of the .claude/ folder

Reference for what each file and folder does, and why it exists. Read this if the user asks for deeper explanation of the structure.

## The top-level files

### CLAUDE.md
The entry point. Committed to git, shared with the team.

**Purpose:** tells Claude everything about the project that would otherwise take an hour of exploring to figure out — the stack, the layout, the conventions, the commands.

**Good CLAUDE.md content:**
- 2–3 sentences on what the project does and why it exists.
- Key commands (install, test, lint, dev server).
- Pointers to rules files.
- Things to avoid (deprecated patterns, legacy code paths).

**Bad CLAUDE.md content:**
- Implementation tutorials (that belongs in the code or a README).
- Generic advice like "write clean code" (useless).
- Stale information (update it when the project changes).

### CLAUDE.local.md
Personal overrides. Not committed (in `.gitignore`).

**Purpose:** give each developer a place to put preferences and notes without polluting the team-shared config.

**Good CLAUDE.local.md content:**
- "I use pyenv with Python 3.12."
- "I'm working on the auth refactor, pick up where I left off."
- "Don't write tests when I say 'quick fix'."

## The .claude/ folder

### settings.json / settings.local.json
Permissions and tool configuration. **Not generated by this skill** — Claude Code manages these itself via `/init` and the permissions UI. If the user wants to edit them, direct them to those interfaces.

### commands/
Custom slash commands. E.g., `commands/review.md` makes `/project:review` available.

**Not generated by this skill** — slash commands are too project-specific to template usefully. If the user wants one, help them write it as a one-off.

### rules/
Modular convention files. Each file covers one topic in depth.

**Why modular?** So you can reference specific files from specific parts of CLAUDE.md ("for testing conventions, see rules/testing.md"), and so you can iterate on one area without rewriting the whole config.

**Typical files:**
- `code-style.md` — naming, formatting, language-specific conventions.
- `testing.md` — how to write tests, what to test, what frameworks.
- `api-conventions.md` — URL structure, error formats, versioning.

Add more as the project grows: `commit-style.md`, `database.md`, `frontend.md`, etc.

### skills/
Auto-invoked workflows. Each skill is a folder with a `SKILL.md` that has YAML frontmatter (`name`, `description`) and a markdown body.

**When to use a skill:** a repeatable workflow that Claude should trigger automatically in specific contexts. E.g., "when touching auth code, run the security review checklist."

**When not to use a skill:** one-off tasks, or things that are better as regular documentation in `rules/`.

### agents/
Specialized sub-agent roles. Each is a markdown file with YAML frontmatter (`name`, `description`, optional `tools`, optional `model`) plus a system prompt that defines the role.

**Why agents?** Context isolation and specialization. Each agent runs in its own context window with its own tool permissions — so a reviewer can't accidentally edit code, and an implementer isn't polluted by the exploratory reads a planner did.

**The pipeline pattern (minimal, 4 agents).** The default bundle is designed around community-validated sweet spot of 3–5 agents:

```
      Plan                Do                   Check
 ┌──────────────┐   ┌──────────────┐   ┌──────────────┐
 │              │─▶ │              │─▶ │              │
 │   planner    │   │ implementer  │   │   reviewer   │
 │              │   │              │   │              │
 └──────────────┘   └──────────────┘   └──────────────┘
      │                   ▲                    │
      │                   │                    │
      ▼                   │                    ▼
   [writes           [human approves     [issues found?]
    PLAN.md /         the plan]                │
    FIX_PLAN.md]                               ▼
                                       back to planner
                                        with findings

        ┌──────────────┐
        │  researcher  │   ← Parallel track. Dispatch any time
        │              │     the task needs lots of file reads
        └──────────────┘     or external lookups.
```

**Why only 4 agents?** Community evidence is consistent: Anthropic recommends 3–5 for most workflows; John Kim's 30-tips article calls "more than three" overkill; PubNub's production pipeline runs on 3 stages. Tokens scale linearly with agent count — more agents means more token cost without proportional benefit. This skill starts minimal; users can always split `reviewer` later if they have a concrete need (e.g., a dedicated `prompt-reviewer` for AI-heavy projects).

**The separation of plan from do** is the whole point. The `planner` self-reviews its plan (four-angle check: scope / technical soundness / completeness / risk) before handing off to the human. Only after human approval does `implementer` execute. This gives you:

- A plan-stage quality gate (self-review + human approval) — much cheaper than fixing code later
- An implementer that's not second-guessing scope while it codes
- A durable spec (`PLAN.md`) that survives context boundaries
- Plans you can iterate on by editing the `.md`, not re-prompting

**Design principles from community best practices:**
- **Start minimal.** 3–5 agents is the sweet spot. Start with 4; split only when you have a concrete pain point.
- **Single responsibility per agent.** Don't stack unrelated jobs.
- **Action-oriented descriptions.** Trigger phrases like "Use PROACTIVELY when..." help Claude auto-delegate.
- **Scoped tools.** `planner`, `reviewer`, `researcher` are read-only (`Read, Grep, Glob, Bash`). Only `implementer` gets `Write`/`Edit`.
- **Model selection.** `sonnet` for most; `opus` for heaviest reasoning; `haiku` for simple scans.

The 4 default agents map cleanly to the canonical phases of any engineering task: **plan / do / check / research**.

## The git workflow

```bash
# Things to commit:
git add CLAUDE.md .claude/ .gitignore
git commit -m "Add Claude Code configuration"

# Things that stay local:
# - CLAUDE.local.md (gitignored)
# - .claude/settings.local.json (gitignored)
```

## Evolving the folder

Start minimal. Add files as real needs emerge:
- A pattern keeps coming up wrong? → add a rule.
- A workflow keeps repeating? → add a skill.
- You keep wishing for an independent reviewer? → add an agent.

Resist adding speculative structure. Unused files are just noise that dilutes the useful signal.
